代码调试

  • 调试
PHP

在敏捷开发大行其道的今天，代码调试是能否提高工作效率的一个重要问题，在我们平时的软件开发中，代码调试所花费的时间要远高于coding的时间，因此一个高效的代码调试姿势是我们所需要的。

写在之前

Framework Interoperability Group（框架可互用性小组）简称FIG，成立于 2009年。FIG最初由几位知名PHP框架开发者发起，在吸纳了许多优秀的大脑和强健的体魄后，提出了 PSR-1 到 PSR-4 四套 PHP 非官方规范：

  1. PSR-1 (Basic Coding Standard) 基础编码标准
  2. PSR-2 (Coding Style Guide) 编码风格向导
  3. PSR-3 (Logger Interface) 日志接口
  4. PSR-4 (Improved Autoloading) 自动加载优化标准
    // PSR-0 (Autoloading Standard) 自动加载标准 - 截止到2014年10月21日，PSR-0已被弃用。 推荐替代使用PSR-4。

基于此套标准，composer横空出世，利用PHP 5.3版出现的命名空间，以及PSR-0，PSR-4标准，在此之上构造了一个繁荣的PHP生态系统，它类似于javascript的NPM，python的PIP，ruby的gem，通俗的说这是一个php自己的第三方包管理系统。

言归正传，个人对于代码调试的一些体会

我认为代码调试主要分为三个阶段.

  1. 开发时的代码调试
  2. 开发后的测试
  3. 程序运行时产生的异常

开发时如何进行代码调试

就PHP来说，在开发时我们需要知道的信息大致有以下几个

  • 某个变量的值和类型
  • 程序运行到了哪一步
  • 运行完一段代码所花费的时间
    为了知道这些信息，一般不想折腾的时候比较接地气的做法是在程序中直接将变量输出到页面中，这种做法非常的简单粗暴，优点在于方便快捷，能快速的定位BUG，缺点在于事后清理代码十分的麻烦，并且如果不慎有未清理掉的调试函数在代码中，将会影响到程序的运行。推荐一些比较高效的工具来帮助我们更好的进行代码调试

注：以下所有工具均使用composer安装

开发完成时如何进行单元测试

PHPUnit

单元测试有几个非常直观的好处

  • 单元测试可以无限次被执行
  • 单元测试是很好的文档
  • 单元测试可以很快地被执行
  • 单元测试可以被任何人和系统执行
  • 单元测试可以快速定位到错误代码
  • 单元测试不会突然被女友喊回家

就概念而言，单元测试是对一个模块中最基础的逻辑单元进行测试

假设现在你需要重构一个已经离职多年的程序员遗留下的代码，而你不能保证你的程序是否完全依照正确的业务逻辑执行，而此时你发现了以前的程序员对于这段业务代码的一个单元测试用例，你用你重构后的代码可以跑过这个测试用例，这个时候心里就会特别有底气，单元测试最大的好处是这个单元如果被修改了以后，通过跑测试就能发现是否会影响到原有的逻辑，是否和原来设计的预期保持一致

因此单元测试一定要由这段代码的开发者来编写，因为只有开发者最清楚这段代码的业务实现
